1) Purpose & Scope

This document justifies Minishell both functionally and technically for a 42 School evaluation. It is also designed as a learning guide to understand the project end-to-end.

Functional Goal
Provide a POSIX-like command interpreter with core shell behavior (parsing, execution, pipes, redirections, builtins).
Technical Goal
Design a robust lexer/parser/executor pipeline in C with proper memory, signal, and process management.
Pedagogical Goal
Make shell internals understandable through examples, flow explanations, and a small interactive parser demo.

2) Functional Specification

Supported behavior

  • Prompt loop, command reading, and execution.
  • Builtins: echo, cd, pwd, export, unset, env, exit.
  • Pipes (|), redirections (<, >, >>, <<).
  • Environment expansion ($VAR, $?), quote handling, and signal behavior.
  • Bonus behavior in this repository: logical operators &&/||, parentheses grouping, wildcard *.

Command examples

echo "hello minishell"
export USERNAME=student
cat infile | grep minishell > outfile
false || echo "fallback"
(cd /tmp && pwd) | cat

Non-functional requirements

  • Reliability: predictable exit statuses and robust syntax error reporting.
  • Correctness: behavior aligned with minishell subject expectations.
  • Code quality: modular source tree (lexer/parser/expander/executor/builtins).
  • Portability note: depends on readline headers/libraries in local environment.

3) Technical Architecture

Processing pipeline:

Readline input
  -> Lexer (tokens)
  -> Parser (command/AST-like structures)
  -> Expander (variables, quote-aware string handling)
  -> Executor (builtins, fork/execve, redirections, pipes)
  -> Status code update and loop

Lexer

Transforms input into token stream while respecting quote state and special operators.

Parser

Creates command structures and redirection links; bonus parser extends with &&, ||, and grouping.

Executor

Runs builtins in-process when needed, external commands via fork + execve, orchestrates pipelines and FDs.

FD and process model (why it matters)

Pipes and redirections require precise file descriptor duplication (dup2) and closure order. A wrong order can deadlock pipelines or leak descriptors.

Signal strategy (interactive shell expectations)

Main loop and child processes usually need different signal handling policies. The project’s signal module centralizes this behavior.

Error-handling philosophy

Report syntax/runtime errors clearly, keep shell alive when possible, and always preserve a meaningful last exit status for $?.

4) 42 Evaluation Mapping

Use this quick checklist during defense:

  • Can you explain the full input-to-execution flow without looking at code?
  • Can you justify where and why builtins run in parent vs child process?
  • Can you walk through one pipeline command and one heredoc command?
  • Can you explain quote behavior and variable expansion edge cases?
  • Can you show one syntax error path and status code behavior?

Defense tip: start with architecture, then a concrete command trace, then edge cases (signals, quotes, redirections).

5) Learning Path (Fast & Practical)

  1. Map the folders: identify one responsibility per module.
  2. Single-command trace: follow echo $USER through lexer → parser → expander → executor.
  3. Pipe trace: follow cat file | grep x, documenting each process and fd.
  4. Redirection trace: compare > vs >> open flags and behavior.
  5. Bonus logic trace: prove short-circuit behavior of &&/||.

Repeat traces until you can explain behavior from memory in less than 2 minutes each.
