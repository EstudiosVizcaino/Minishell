1. What is Minishell?

Minishell is a UNIX command-line interpreter written in C, built as part of the 42 School curriculum. It mimics the essential behaviour of bash: it reads commands you type, understands their structure, replaces variables with their values, and finally runs the programs you requested.

Think of it as a simplified version of the terminal you use every day. When you type:

minishell> echo "Hello $USER" | tr a-z A-Z > output.txt

Minishell must understand that you want to:

  1. Run echo with argument Hello <your username>.
  2. Pipe that output to tr which converts lowercase to uppercase.
  3. Save the final result to output.txt instead of printing it.

This seemingly simple task requires parsing, variable expansion, process creation, pipe setup, and file redirection — all implemented from scratch in C.

📄 Language

C (C99, POSIX)

📚 Library

readline, libft (custom)

🔧 Key syscalls

fork, execve, pipe, dup2, waitpid, sigaction

3. Data Structures (the building blocks)

Before diving into each stage, you need to know the data types that travel through the pipeline. All of them are defined in includes/minishell.h.

3.1 t_token — a single lexical unit

typedef struct s_token
t_token_typetypeWhat kind of token: WORD, PIPE, REDIR_IN, REDIR_OUT, HEREDOC, APPEND, AND, OR, LPAREN, RPAREN, EOF
char *valueThe raw text, e.g. "echo" or "|" or ">>"
struct s_token *nextNext token in the singly-linked list

Example: lexing echo hi | cat produces the chain:
[WORD:"echo"] → [WORD:"hi"] → [PIPE:"|"] → [WORD:"cat"] → NULL

3.2 t_redir — one I/O redirection

typedef struct s_redir
t_token_typetypeREDIR_IN (<), REDIR_OUT (>), HEREDOC (<<), APPEND (>>)
char *fileThe target filename or heredoc delimiter
intheredoc_fdRead-end of the pipe used to pass heredoc content (or -1)
struct s_redir *nextNext redirection in the list

3.3 t_cmd — one command node

typedef struct s_cmd
char **argsNULL-terminated argument array: args[0] = command name, args[1..n] = arguments
t_redir *redirsLinked list of redirections attached to this command

3.4 t_ast — a node in the Abstract Syntax Tree

typedef struct s_ast
t_node_typetypeNODE_CMD, NODE_PIPE, NODE_AND, NODE_OR
t_cmd *cmdCommand data (only for NODE_CMD nodes)
struct s_ast *leftLeft child (e.g. left side of a pipe)
struct s_ast *rightRight child (e.g. right side of a pipe)

Example: ls | grep c builds this tree:

      NODE_PIPE
      /       \
  NODE_CMD  NODE_CMD
  cmd="ls"  cmd="grep c"

3.5 t_env — one environment variable

typedef struct s_env
char *keyVariable name, e.g. "PATH"
char *valueVariable value, e.g. "/usr/bin:/bin"
struct s_env *nextNext variable in the linked list

3.6 t_shell — the global shell state

typedef struct s_shell
t_env *envHead of the environment linked list
intlast_exitExit code of the last command ($?)
char *inputCurrently unused storage slot
intin_heredocFlag set while reading a heredoc body
t_shell is the “spine” of the program. It is passed by pointer to almost every function so they can read/update environment variables and the last exit status.

4. Stage 0: Startup & Environment Initialisation

File: srcs/main.c, srcs/env/env.c

4.1 Entry point: main()

int main(int argc, char **argv, char **envp)
{
    t_shell shell;

    shell.env       = env_init(envp);   // convert envp[] into linked list
    shell.last_exit = 0;               // $? starts at 0
    shell.input     = NULL;
    shell.in_heredoc = 0;
    main_loop(&shell);                  // enter infinite REPL
    env_free(shell.env);               // clean up on exit
    rl_clear_history();                // free readline history
    return (shell.last_exit);
}

4.2 Converting envp into a linked list (env_init)

The OS gives us environment variables as char **envp, an array of strings like "PATH=/usr/bin:/bin". We turn each one into a t_env node:

// "PATH=/usr/bin:/bin" → { key="PATH", value="/usr/bin:/bin" }
t_env *env_parse_entry(char *entry)  // splits on first '='
t_env *env_init(char **envp)          // calls env_parse_entry for each

This linked list makes it easy to look up variables, add new ones, remove them, and convert back to char ** for execve.

5. Stage 1: readline — Reading User Input

File: srcs/main.c — function main_loop()

static void main_loop(t_shell *shell)
{
    char *line;

    while (1)
    {
        setup_signals();                    // CTRL+C shows new prompt, CTRL+\ ignored
        line = readline("minishell> ");   // blocking: waits for user to press Enter
        if (!line)                          // NULL = CTRL+D (EOF) → exit
        {
            ft_putstr_fd("exit\n", STDOUT_FILENO);
            break;
        }
        if (*line)
            add_history(line);              // non-empty lines go into history (↑ key)
        run_line(line, shell);              // process the command
        free(line);
    }
}
readline is a library that provides line-editing (arrow keys, backspace, Ctrl+A/E…) and command history for free. Without it we would have to reimplement all that manually.

What happens on CTRL+D?

readline returns NULL when it reaches EOF (no more input). That signals the shell to exit gracefully — just like real bash.

What happens on CTRL+C?

The signal handler (sig_handler) calls rl_on_new_line() and rl_redisplay() to print a new blank prompt — same behaviour as bash.

The lexer receives the raw string the user typed and converts it into a linked list of tokens. A token is the smallest unit with a meaningful identity: a word, a pipe, a redirection operator, etc.

    // Append resulting token to linked list
    // Return head of list

6.2 Operator tokens (handle_operator)

When the current character is |, <, or >, the lexer checks the next character too to catch double-char operators:

Chars seenToken typeValue
|TOKEN_PIPE|
<TOKEN_REDIR_IN<
>TOKEN_REDIR_OUT>
<<TOKEN_HEREDOC<<
>>TOKEN_APPEND>>

6.3 Word tokens (handle_word)

Everything that is not an operator and not whitespace is a “word”. The tricky part is quoted text: inside quotes, operators and spaces lose their special meaning.

// word_end() advances index until it hits whitespace or an operator,
// but if it encounters a quote, it skips over the ENTIRE quoted segment
// (including its closing quote) before continuing.
//
// Example: echo "hello world" | cat
//                       ↑ the space inside quotes is NOT a word boundary
// Result:  [WORD:"echo"] [WORD:'"hello world"'] [PIPE:"|"] [WORD:"cat"]
Important: at the lexer stage, quote characters are kept as part of the word value. They are removed later by the expander when it processes the string character by character.

7. Stage 3: Parser — Building the Abstract Syntax Tree

Files: srcs/parser/parser_entry.c, parser.c, parser_cmd.c, parser_redir.c

The parser walks the token list and builds an Abstract Syntax Tree (AST) — a tree that represents how the commands relate to each other (are they piped? sequential? etc.).

7.1 Entry point: parser()

t_ast *parser(t_token *tokens)
{
    t_token *tok = tokens;
    if (!tok) return (NULL);
    return (parse_pipeline(&tok));   // grammar entry point
}

7.2 Grammar (simplified)

pipeline  = command ( '|' command )*
command   = ( WORD | redir )*
redir     = ( '<' | '>' | '<<' | '>>' ) WORD

7.3 parse_pipeline() — building pipe chains

t_ast *parse_pipeline(t_token **tokens)
{
    left = parse_command(tokens);          // parse left side
    if (current token == TOKEN_PIPE)
    {
        advance past '|';
        pipe_node = new_ast_node(NODE_PIPE); // create pipe node
        pipe_node->left  = left;
        pipe_node->right = parse_pipeline(tokens); // recursive! handles a|b|c
        return pipe_node;
    }
    return left;  // no pipe, just a single command
}
This is recursive descent parsing. parse_pipeline calls itself for the right side of each pipe, so a | b | c becomes a right-leaning tree: PIPE(a, PIPE(b, c)).

7.4 parse_command() — collecting args and redirections

For each command, the parser does two passes over the token stream:

  1. Count how many WORD tokens there are (to allocate args array).
  2. Fill: word tokens go into cmd->args[], redirection tokens go into the cmd->redirs list.
// Input tokens for:  cat < in.txt file.txt
// count_word_tokens → 2  ("cat", "file.txt")
// fill_cmd builds:
//   cmd->args    = ["cat", "file.txt", NULL]
//   cmd->redirs  = { type=REDIR_IN, file="in.txt", next=NULL }

7.5 Resulting AST for ls -l | grep ".c" > out.txt

            NODE_PIPE
           /         \
      NODE_CMD       NODE_CMD
      args=["ls",    args=["grep", ".c"]
            "-l"]    redirs={ REDIR_OUT, "out.txt" }
After parsing the token list is freed (it is no longer needed). Only the AST lives on.

8. Stage 4: Expander — Variables, Quotes & Wildcards

Files: srcs/expander/expander.c, expander_args.c, expander_str.c, expander_utils.c

The expander walks the AST and transforms every argument and redirection filename by:

  • Replacing $VAR with its environment value.
  • Replacing $? with the exit code of the last command.
  • Removing quote characters (' and ") from the final value.

8.1 Entry: expand_ast()

void expand_ast(t_ast *ast, t_shell *shell)
{
    if (ast->type == NODE_CMD)
        expand_cmd(ast->cmd, shell);   // process this command's args + redirs
    else
    {
        expand_ast(ast->left,  shell); // recurse into left child (e.g. left of pipe)
        expand_ast(ast->right, shell); // recurse into right child
    }
}

8.2 Expanding one string: expand_str()

This function walks a string character by character and handles four cases:

Character(s) encounteredAction
'...' single quotesCopy text verbatim; NO variable expansion inside
"..." double quotesExpand $VAR inside, but spaces are preserved as-is
$VAR outside quotesLook up VAR in environment, substitute its value
Any other characterCopy it as-is
 
// Example: expand_str('"Hello $USER"', shell) where USER=alice
// → enters expand_double_quote()
// → sees "$USER", calls expand_var() → "alice"
// → result: "Hello alice"   (quotes removed, variable expanded)

// Example: expand_str("'$HOME'", shell)
// → enters expand_single_quote()
// → result: "$HOME"          (literally, no expansion)

8.3 Variable expansion: expand_var()

char *expand_var(char *str, int *i, t_shell *shell)
{
    (*i)++;                          // skip '$'
    if (str[*i] == '?')             // special case: $?
        return ft_itoa(shell->last_exit);
    // read alphanumeric/underscore chars as variable name
    name = str[start .. *i];
    val  = env_get(shell->env, name); // look up in environment linked list
    return val ? val : "";           // unset variable expands to empty string
}

8.4 Handling empty variable expansions

If an argument consists only of variable references that all expand to empty string (e.g. $UNSET_VAR), the whole argument is removed from cmd->args. This matches bash behaviour.

9. Stage 5: Executor — Running Commands

Files: srcs/executor/executor_run.c, executor_cmd.c, executor_pipe.c, executor_redir.c, executor_utils.c, executor.c

9.1 Entry: execute()

int execute(t_ast *ast, t_shell *shell)
{
    if (ast->type == NODE_PIPE) return exec_pipe(ast, shell);
    if (ast->type == NODE_CMD)  return exec_cmd(ast, shell);
    return (0);
}

9.2 Single command: exec_cmd()

The executor handles four distinct cases for a single command node:

Case 1: Empty command with redirections

e.g. > out.txt — just applies the redirections (creates/truncates the file) with no program to run.

Case 2: Completely empty

No arguments, no redirections — returns 0 immediately.

Case 3: Builtin command

e.g. cd /tmp. Builtins must run in the parent process so they can modify shell state. The FDs are temporarily saved, redirections applied, builtin runs, then original FDs are restored.

Case 4: External command

e.g. ls -la. fork() creates a child; the child applies redirections, finds the executable in PATH, then execve() replaces itself with the target program. The parent waitpid()s for the child.

9.3 External command execution flow

// In exec_cmd() for external commands:
pid = fork();
if (pid == 0)              // ←── CHILD PROCESS
{
    setup_signals_child(); // reset SIGINT/SIGQUIT to defaults
    apply_redirs(cmd->redirs); // rewire stdin/stdout if needed
    path = find_executable(cmd->args[0], shell->env);
    // search PATH: "/usr/bin/ls" for "ls"
    envp = env_to_array(shell->env); // rebuild char** for execve
    execve(path, cmd->args, envp);  // REPLACE child with target program
    // execve only returns on error
    exit(127);                      // command not found
}
// ←── PARENT PROCESS: wait for child
return wait_for_child(pid);  // returns child's exit status

9.4 Finding the executable: find_executable()

// If name contains '/', use it directly (absolute/relative path)
if (ft_strchr(name, '/')) return ft_strdup(name);

// Otherwise split PATH on ':' and try each directory:
// PATH=/usr/bin:/bin → try "/usr/bin/ls", then "/bin/ls"
// access(candidate, X_OK) checks if file is executable

9.5 Builtin with redirections: exec_builtin_redir()

saved_in  = dup(STDIN_FILENO);   // save original stdin
saved_out = dup(STDOUT_FILENO);  // save original stdout
apply_redirs(cmd->redirs);        // redirect as requested
ret = exec_builtin(cmd, shell);   // run the builtin (may write to stdout)
dup2(saved_in,  STDIN_FILENO);    // restore stdin
dup2(saved_out, STDOUT_FILENO);   // restore stdout
close(saved_in);  close(saved_out);
Why save/restore FDs? If cd has a > file redirection, its output goes to that file. But after cd finishes, the shell must print to the terminal again, so we restore the original FDs.

10. Builtin Commands

Files: srcs/builtins/

Builtins are commands implemented inside the shell rather than as external programs. They are special because they may need to modify the shell’s own state (working directory, environment variables, etc.).

BuiltinWhat it doesWhy it must be a builtin
echo [-n] Prints arguments to stdout. -n skips the trailing newline. Could be external, but 42 requires implementation.
cd <path> Changes the working directory; updates PWD and OLDPWD. chdir() only affects current process. An external cd in a child would not change the shell’s directory.
pwd Prints the current working directory. Must reflect cd changes that live in the parent process.
export [KEY=VAL] Sets or displays environment variables. Must modify the shell’s own t_env list (parent state).
unset KEY Removes an environment variable. Same as export — modifies parent environment.
env Prints all environment variables. Must read the shell’s own t_env list.
exit [n] Exits the shell with status n. Must terminate the shell process itself.

Builtin detection: is_builtin()

int is_builtin(char *name)
{
    return (ft_strcmp(name, "echo") == 0
         || ft_strcmp(name, "cd")   == 0
         || ft_strcmp(name, "pwd")  == 0
         || ft_strcmp(name, "export") == 0
         || ft_strcmp(name, "unset") == 0
         || ft_strcmp(name, "env")  == 0
         || ft_strcmp(name, "exit") == 0);
}
Why does cd update OLDPWD?

OLDPWD is a standard POSIX variable that stores the previous working directory. This lets users do cd - (not implemented here, but standard shells support it). Minishell stores OLDPWD before changing directory so the environment stays consistent.

11. Signal Handling

File: srcs/signals/signals.c

Signals are asynchronous notifications sent to a process. Minishell must handle them differently depending on context:

ContextCTRL+C (SIGINT)CTRL+\ (SIGQUIT)
Interactive shell (no command running) Print newline, show new prompt Ignored
While a child command is running Default behaviour (kills child process) Default behaviour (core dump in child)
Inside a heredoc prompt Close stdin to abort heredoc input Ignored

Global signal flag: g_signal

extern int g_signal;  // set to signal number by signal handler
                      // allows main loop to detect async signals
Only one global variable is allowed by 42 Norminette rules. g_signal is the only global, and it is used solely to communicate between the async signal handler and the synchronous main loop.

Three signal configurations

setup_signals()         // interactive: SIGINT → sig_handler; SIGQUIT ignored

13. Pipes in Depth

File: srcs/executor/executor_pipe.c

A pipe (|) connects the stdout of one command to the stdin of the next. The OS provides a pipe() syscall that creates two file descriptors: a write end and a read end.

Step-by-step: executing ls | grep c

1. Create the pipe
int pipefd[2];
pipe(pipefd);   // pipefd[0]=read end, pipefd[1]=write end
2. Fork the LEFT child (ls)
pid1 = fork();
if (pid1 == 0)   // child 1 = "ls"
{
    dup2(pipefd[1], STDOUT_FILENO); // stdout → write end
    close(pipefd[0]); close(pipefd[1]);
    execute(ast->left, shell);      // run "ls"
    exit(...);
}
3. Fork the RIGHT child (grep c)
pid2 = fork();
if (pid2 == 0)   // child 2 = "grep c"
{
    dup2(pipefd[0], STDIN_FILENO);  // stdin ← read end
    close(pipefd[0]); close(pipefd[1]);
    execute(ast->right, shell);     // run "grep c"
    exit(...);
}
4. Parent closes pipe ends & waits
close(pipefd[0]); close(pipefd[1]); // parent doesn't use the pipe!
waitpid(pid1, ...);
waitpid(pid2, ...);
Critical: if the parent does NOT close its copies of pipefd[0] and pipefd[1], the read end of the pipe will never reach EOF (because the parent still has the write end open), causing grep to hang forever. Always close unused pipe ends!

File descriptor picture

PARENT process:
  pipefd[0] (read)  ───closed───
  pipefd[1] (write) ───closed───

CHILD 1 (ls):
  stdout (fd 1)  ──────────→  pipefd[1] (write end)
  pipefd[0], pipefd[1] both closed

CHILD 2 (grep):
  stdin  (fd 0)  ←──────────  pipefd[0] (read end)
  pipefd[0], pipefd[1] both closed

14. Redirections in Depth

File: srcs/executor/executor_redir.c

Redirections change where a command reads from or writes to.

OperatorEffectHow it works
< file Read stdin from file open(file, O_RDONLY) then dup2(fd, STDIN_FILENO)
> file Write stdout to file (truncate) open(file, O_WRONLY|O_CREAT|O_TRUNC, 0644) then dup2(fd, STDOUT_FILENO)
>> file Append stdout to file open(file, O_WRONLY|O_CREAT|O_APPEND, 0644) then dup2(fd, STDOUT_FILENO)
<< DELIM Read stdin from inline text (heredoc) See section 15

dup2 explained

dup2(fd, STDOUT_FILENO);
// Makes file descriptor STDOUT_FILENO (which is 1) point to the same
// file/pipe as fd. After this call, anything the program writes to
// stdout actually goes to "fd"'s target.
close(fd);  // we no longer need the original fd

Multiple redirections

apply_redirs() walks the t_redir linked list and applies them in order. The last redirection of the same type wins (same as bash):

cat < a.txt < b.txt   // cat reads from b.txt (last wins)
echo hi > x > y      // output goes to y (last wins)

15. Here-doc in Depth

A here-doc lets you type multi-line input inline in the shell prompt. Example:

minishell> cat << EOF
> line one
> line two
> EOF
line one
line two

How open_heredoc() works

1. Create an anonymous pipe
int pipefd[2];
pipe(pipefd); // pipefd[1]=write, pipefd[0]=read
2. Read lines from the user with a “> ” prompt
while (1) {
    line = readline("> ");
    if (!line || ft_strcmp(line, redir->file) == 0)
        break;               // stop when delimiter reached or CTRL+D
    write(pipefd[1], line, ft_strlen(line)); // write to pipe
    write(pipefd[1], "\n", 1);
    free(line);
}
3. Close write end; store read end
close(pipefd[1]);
redir->heredoc_fd = pipefd[0]; // saved for later use
4. When the command runs, apply_redirs() connects the read end to stdin
dup2(redir->heredoc_fd, STDIN_FILENO);
close(redir->heredoc_fd);
Why a pipe instead of a temp file? A pipe avoids disk I/O, needs no cleanup, and is automatically destroyed when both ends are closed. The content lives only in kernel memory until read.

16. Bonus Features

Files: srcs_bonus/

16.1 Logical operators: && and ||

These extend the grammar beyond pipelines:

grammar (bonus):
  and_or   = pipeline ( ('&&' | '||') pipeline )*

The executor for NODE_AND/NODE_OR in exec_and_or():

cmd1 && cmd2  // run cmd2 only if cmd1 exits 0
cmd1 || cmd2  // run cmd2 only if cmd1 exits non-0

16.2 Parentheses () for grouping

(cd /tmp && ls) | grep c
// everything in () is treated as a sub-expression
// parsed as a sub-pipeline in parse_and_or_bonus()

16.3 Wildcard expansion *

When an argument contains *, the expander calls expand_wildcard():

1. Open current directory with opendir(".")
2. For each entry, call match_wildcard(pattern, entry_name)

The matcher recursively handles * (matches any sequence of chars) vs. literal characters.

3. Collect all matches; sort them (alphabetical order)
4. Replace the single *.c argument with the list of matches

If no match → keep the original pattern (bash behaviour).

17. Full Command Trace

Let’s follow the complete execution of:

export MSG="hello world" && echo $MSG | tr a-z A-Z > out.txt

(Bonus mode, with &&)

① readline returns the raw string
"export MSG=\"hello world\" && echo $MSG | tr a-z A-Z > out.txt"
② Lexer produces 14 tokens
[WORD:"export"] [WORD:"MSG=\"hello world\""]
[AND:"&&"]
[WORD:"echo"] [WORD:"$MSG"]
[PIPE:"|"]
[WORD:"tr"] [WORD:"a-z"] [WORD:"A-Z"]
[REDIR_OUT:">"] [WORD:"out.txt"]
③ Parser builds the AST
          NODE_AND
         /        \
   NODE_CMD       NODE_PIPE
   args=["export",          /          \
         "MSG=hello world"] NODE_CMD    NODE_CMD
                          args=["echo","$MSG"]
                                      args=["tr","a-z","A-Z"]
                                      redirs={REDIR_OUT,"out.txt"}
④ Expander resolves variables
  • In export node: the argument MSG="hello world" is kept as-is (export handles it).
  • In echo node: $MSG → "" (MSG not yet set at expansion time; export has not run yet!).
  • In tr node: no variables, stays the same.
This is a nuance: expansion happens before any command runs. So if export sets MSG and echo $MSG expects to see it, you would normally separate them across two lines or use a subshell trick.
⑤ Executor runs NODE_AND
  1. Runs left child: export MSG="hello world" → adds MSG to shell.env, returns 0.
  2. Because exit code is 0 (&&), runs right child: the pipeline.
⑥ Executor runs NODE_PIPE
  1. pipe(pipefd) creates the data channel.
  2. Forks child 1: runs echo "" (stdout → pipe write end).
  3. Forks child 2: runs tr a-z A-Z (stdin ← pipe read end, stdout → out.txt).
  4. Parent closes pipe ends, waits for both children.
⑦ File out.txt is created with the result

Content: an empty line (since $MSG expanded to empty). tr converts nothing, exit code 0. Shell prompt returns.

⑧ AST and tokens are freed, loop repeats

free_ast() recursively frees all nodes. shell.last_exit = 0. readline() is called again.

19. Glossary

TermPlain-language definition
ShellA program that reads commands from a user and executes them.
TokenThe smallest meaningful piece of a command: a word, a pipe, a redirection symbol.
Lexer / TokenizerThe stage that splits raw text into tokens.
ParserThe stage that arranges tokens into a structured execution plan (AST).
ASTAbstract Syntax Tree — a tree where each node is either a command or an operator connecting two sides.
ExpanderThe stage that replaces $VAR and removes quote characters.
ExecutorThe stage that actually runs programs using OS syscalls.
BuiltinA command implemented inside the shell itself, not an external program.
fork()A syscall that creates an exact copy of the current process (“child”).
execve()A syscall that replaces the current process with a different program.
pipe()A syscall that creates a data channel between two file descriptors.
dup2()A syscall that makes one file descriptor point to the same target as another.
waitpid()A syscall that makes the parent process wait for a child to finish and retrieve its exit code.
File descriptor (FD)A small integer the OS uses to identify an open input/output resource. 0=stdin, 1=stdout, 2=stderr.
RedirectionChanging where a command reads from or writes to (file instead of terminal).
HeredocA way to provide multi-line input directly in the terminal until a delimiter line is reached.
SignalAn asynchronous OS notification to a process (e.g., CTRL+C sends SIGINT).
Environment variableA named string value inherited by child processes (e.g., PATH, HOME, USER).
Exit status / $?A number (0–255) a program returns when it finishes. 0 = success, non-zero = error.
PATHAn environment variable listing directories where the shell looks for executable programs.
WildcardA special character (*) that matches multiple filenames.
Linked listA data structure where each element holds a pointer to the next one.
