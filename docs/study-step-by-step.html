Minishell: Very Detailed Step-by-Step Guide

This file is an additional study document focused on tiny, explicit steps. It follows one full example from typing to process execution, and also lists other supported options.

0) What this project is

Minishell is a small shell (command interpreter). It reads your line, understands its syntax, expands variables, and executes builtins or external programs while handling pipes/redirections.

Main flow in this repository: 

readline -> lexer -> parser -> expander -> executor -> exit status saved -> loop again

1) Files and modules (where each stage lives)

  • srcs/main.c - prompt loop and high-level orchestration.
  • srcs/lexer - tokenization.
  • srcs/parser - syntax structure and command nodes.
  • srcs/expander - $VAR and $? expansion.
  • srcs/executor - builtins, forks, pipes, redirections, command run.
  • srcs/builtins - builtin commands like cd, export.
  • srcs/signals - signal behavior.
  • srcs_bonus - bonus logic: &&, ||, (), wildcard.

2) Worked example used for this walkthrough

We will follow this exact line:

export LOG=error && cat app.log | grep "$LOG" > errors.txt

Why this is a good example: it touches builtin execution, logical AND, pipeline, quote-aware expansion, and output redirection.

3) Micro-steps: from keyboard input to prompt return

Step 1 - Prompt is waiting

Program is inside the main loop and waits for user input through readline.

Step 2 - User types characters

Every character is read as part of one raw string. Enter finishes the line.

Step 3 - Raw string handed to lexer

Lexer scans left to right and groups characters into tokens, preserving quote context.

[export] [LOG=error] [&&] [cat] [app.log] [|] [grep] ["$LOG"] [>] [errors.txt]

Step 4 - Parser validates grammar and builds structure

Parser checks operator placement and creates executable relations:

  • Left side command: export LOG=error
  • Connector: && (execute right side only if left side succeeds)
  • Right side pipeline: cat app.log | grep "$LOG"
  • Final redirection: > errors.txt applied to pipeline output

Step 5 - Expander resolves variables

"$LOG" becomes "error" using current shell environment.

Important detail: because the variable is in double quotes, it expands but remains one argument.

Step 6 - Executor starts left side of AND

export is a builtin, so it runs in shell logic (not as external binary) and updates shell environment.

Step 7 - AND condition check

If export exit code is 0, right side is executed. Otherwise it is skipped.

Step 8 - Pipeline process preparation

Executor creates a pipe and forks children as needed for cat and grep.

Step 9 - File descriptor wiring

  • cat stdout is connected to pipe write end.
  • grep stdin is connected to pipe read end.
  • Because of > errors.txt, grep stdout is redirected to the file.

Step 10 - External commands executed

Children call exec-family logic to run real binaries found through PATH.

Step 11 - Parent waits and collects status

Parent shell waits for children, computes resulting status, stores it in shell state ($?).

Step 12 - Cleanup + next prompt

Temporary resources are closed/freed. Shell returns to prompt for the next command line.

4) Additional options/features (mentioning even if not used above)

Base features

  • | pipe: send stdout of one command to stdin of next.
  • < input redirection from file.
  • > output redirection (truncate file).
  • >> output append redirection.
  • << heredoc: multiline input until delimiter.
  • Variable expansion: $VAR and status expansion $?.
  • Quotes:
    • Single quotes block expansion.
    • Double quotes allow expansion but preserve grouped text.
  • Builtins: echo, cd, pwd, export, unset, env, exit.

Bonus features (when built with make bonus)

  • && logical AND with short-circuit behavior.
  • || logical OR with short-circuit behavior.
  • () parentheses for grouped command logic/precedence.
  • * wildcard expansion in current directory.

5) Mini mental model for study

1. Read text
2. Split text into meaningful pieces (tokens)
3. Build execution plan from those pieces
4. Resolve dynamic values (variables, status)
5. Run plan with process + FD rules
6. Save status and loop

If you can explain these six points clearly, you already understand the heart of this Minishell project.
