Minishell: Very Detailed Step-by-Step Guide

This file is intentionally extended. It explains the project in tiny steps, then drills into the functions involved in each phase, and iterates one full command walkthrough end-to-end.

0) What this project is (core idea)

Minishell is a small shell (command interpreter). It reads your line, understands its syntax, expands variables, and executes builtins or external programs while handling pipes/redirections. 

readline -> lexer -> parser -> expander -> executor -> update $? -> prompt again

1) Source map (where each phase lives)

  • srcs/main.c and srcs_bonus/main_bonus.c: loop + orchestration.
  • srcs/lexer/*.c (+ srcs_bonus/lexer_utils_bonus.c): lexical analysis.
  • srcs/parser/*.c (+ srcs_bonus/parser*_bonus.c, parse_andor_bonus.c): AST construction + syntax checks.
  • srcs/expander/*.c (+ srcs_bonus/expander*_bonus.c, wildcard files): expansions.
  • srcs/executor/*.c (+ srcs_bonus/executor_bonus.c): command execution.
  • srcs/builtins/*.c: builtin command behavior.
  • srcs/env/*.c: environment linked list utilities.
  • srcs/signals/signals.c: SIGINT/SIGQUIT handling.

2) Worked command we will follow

Bonus walkthrough command:

export LOG=error && cat app.log | grep "$LOG" > errors.txt

This line touches: builtin (export), logical &&, pipeline, variable expansion in quotes, and output redirection.

Note: && is bonus syntax, so this specific line requires make bonus. All individual pieces (export, pipe, quotes, redirection) are still useful to understand the base flow.

4) Ultra-granular flow: from Enter key to next prompt

Step 1 - Prompt is waiting

main() initializes t_shell, then enters main_loop().

In each iteration, setup_signals() is called, then readline("minishell> ") waits.

Step 2 - User types characters

The full line becomes one raw char *. If non-empty, it is pushed with add_history().

Step 3 - run_line() orchestrates pipeline

run_line() performs: lexer -> parser -> expand_ast -> execute, then frees AST.

Step 4 - Lexer generates tokens

lexer() loops over input. It delegates to:

  • is_operator() to classify operator starts
  • handle_operator() to emit tokens like TOKEN_AND, TOKEN_PIPE, redirs
  • handle_word() and word_end() for words and quoted ranges
  • quote_len() to skip complete quoted blocks safely
[export] [LOG=error] [&&] [cat] [app.log] [|] [grep] ["$LOG"] [>] [errors.txt]

Step 5 - Syntax validation (bonus path)

In bonus main: check_syntax() runs before parsing. It rejects invalid operator chains and invalid redirect targets.

Step 6 - Parser builds AST

Base parser entry: parser() -> parse_pipeline().

Bonus parser entry: parser() -> parse_and_or(), which internally uses parse_parens() and parse_pipeline().

Step 7 - parse_command() fills command payloads

parse_command() allocates t_cmd, counts argument words via count_word_tokens(), and fills:

  • args array (duplicated strings)
  • redirs list via make_redir() / add_redir()

Step 8 - AST for our line

NODE_AND
├── NODE_CMD      (export LOG=error)
└── NODE_PIPE
    ├── NODE_CMD  (cat app.log)
    └── NODE_CMD  (grep "$LOG" with redir > errors.txt)

Step 9 - Expander rewrites args and redirs

expand_ast() recursively walks AST and for each command executes:

  • expand_args() for command args
  • expand_redirs() for redirect file targets (except heredoc delimiters)

String-level behavior is done by expand_str(), expand_one(), expand_var(), and join_free().

"$LOG" expands to "error"; single quotes are not expanded.

Step 10 - Bonus wildcard pass (if built in bonus mode)

expand_wildcards_cmd() scans each argument with * and calls expand_wildcard(), merge_args(), and unmask helpers.

expand_wildcard_redir() does the same for redirection targets; multiple matches become "ambiguous redirect".

Step 11 - Executor dispatches by AST node type

execute() (bonus version) dispatches to:

  • exec_and_or() for NODE_AND/NODE_OR
  • exec_pipe() for NODE_PIPE
  • exec_cmd() for NODE_CMD

Step 12 - exec_and_or() short-circuit behavior

exec_and_or() executes left branch first.

  • For &&, if left fails (non-zero), right is skipped.
  • For ||, if left succeeds (0), right is skipped.

For our example, successful export makes the right pipeline execute.

Step 13 - exec_cmd() path for builtin export

exec_cmd() checks builtin via is_builtin(), then uses exec_builtin_redir() so redirections (if any) are handled safely.

exec_builtin() dispatches to builtin_export().

Step 14 - builtin_export() internals

Relevant functions inside builtin_export.c:

  • builtin_export(): loops over export arguments
  • export_one(): splits "KEY=VALUE" or key-only path
  • is_valid_key(): validates identifier format
  • export_with_value(): writes into env list via env_set()
  • print_export(): prints declared vars when no args

Here, LOG=error is inserted/updated in shell->env.

Step 15 - exec_pipe() creates process graph

exec_pipe() creates one OS pipe, forks left and right children:

  • child_left(): dup2(pipefd[1], STDOUT_FILENO), executes left subtree
  • child_right(): dup2(pipefd[0], STDIN_FILENO), executes right subtree
  • wait_children(): waits and returns right-side status

Step 16 - Left command in pipe: cat app.log

exec_cmd() sees non-builtin and forks. Child runs exec_child():

  • setup_signals_child()
  • apply_redirs() (none for this side)
  • find_executable() (PATH lookup)
  • execve() through internal child-run logic

Step 17 - Right command in pipe: grep "$LOG" (runtime arg: "error") > errors.txt

Before command launch, apply_redirs() processes TOKEN_REDIR_OUT with open(...O_TRUNC) and dup2(fd, STDOUT_FILENO).

So grep reads from pipe stdin and writes file output to errors.txt.

Step 18 - Parent waits + status propagation

Parent waits children, computes exit codes (WIFEXITED/WIFSIGNALED), stores final status in shell->last_exit.

Then run loop frees AST/tokens/line and returns to prompt.

6) Additional options/features (including unused ones)

Base features (non-bonus)

  • | pipe: send stdout of one command to stdin of next.
  • < input redirection from file.
  • > output redirection (truncate file).
  • >> output append redirection.
  • << heredoc: multiline input until delimiter.
  • Variable expansion: $VAR and status expansion $?.
  • Quotes:
    • Single quotes block expansion.
    • Double quotes allow expansion but preserve grouped text.
  • Builtins: echo, cd, pwd, export, unset, env, exit.

Bonus features (when built with make bonus)

  • && logical AND with short-circuit behavior.
  • || logical OR with short-circuit behavior.
  • () parentheses for grouped command logic/precedence.
  • * wildcard expansion in current directory.

7) Compact memory checklist

1) readline gets raw line
2) lexer builds token linked list
3) parser converts tokens into AST
4) expander rewrites args/redirs
5) executor walks AST and runs nodes
6) parent stores exit status in shell->last_exit
7) loop returns to prompt

If you can narrate those seven points and name the core functions for each, you understand the project architecture.
